## What broke

### Wide Scan: only `chunk-0.jsonl` produced output

```
auditor/chunks/
  chunk-0.jsonl  ←  50 records scored
  (chunks 1-13 never created)
```

Workflow result: `wide_scan: {chunks: 1, scored: 50, chunk_errors: 0}`. The workflow code thought only 1 chunk existed.

### Root cause: 2MB structured-output truncation

The `wide-scan:load` agent was prompted to read `partner-directory-clean.json` and return `{partners: [...]}` via structured output. The file is:

```
2,086,978 bytes
3,333 records
626 bytes/record on average
→ ~2.0 MB structured output payload
```

That far exceeds the structured-output budget. The agent silently truncated to ~50 records and returned. The schema validator accepted it (the `partners` array had no minLength constraint), so `parallel(...)` ran with `chunks.length === 1` and the workflow continued downstream.

This is a **workflow architecture bug**, not a Wide Scan logic bug. The per-chunk scoring code is fine — the 50 records in `chunk-0.jsonl` are correctly structured (vgps_id, name, weburl, primary_category, subcategories, score, tier, chain_brand, chain_brand_evidence, checks).

### Cascade to Aggregate

Aggregate merged the 50 scored chunk records with the 602 Tier Z records → wrote 652 records to `partners-wide-scan.json`. All the cuts (tier-by-raw, tier-by-canonical, tier-by-strategic, tier-by-subcategory, chain-vs-indie) reflect this partial set. Headline numbers reported:

- `unique_businesses: 643`
- `tier_counts: {A: 5, B: 26, C: 10, D: 9, Z: 602}` ← only ~50 of these are from real scoring; the rest is Tier Z

**Do not treat these cuts as the audit result.** They're a 1.5%-coverage sample.

## The fix — needs Sat's call (two options)

Both options avoid passing large payloads through structured output. Picking either, the Inventory phase stays untouched and re-running is fast (no fetcher restart).

### Option A — index-slice loader, chunks read from disk

- `wide-scan:count` agent returns `{count: 3333}` only (cheap structured output)
- Workflow computes N_CHUNKS and CHUNK_SIZE
- Each chunk agent is told: "Read records [start, end) from `partner-directory-clean.json` and score them." Each chunk agent reads its own slice via Bash/python from disk
- Per-chunk output unchanged (chunks/chunk-N.jsonl)
- **Pro:** no extra files, lean
- **Con:** each chunk agent does its own JSON parse pass (cheap, but 14× redundant)

### Option B — pre-sliced files on disk

- A new `wide-scan:slice` agent (or Bash step) splits `partner-directory-clean.json` into 14 files: `auditor/slices/slice-0.json` through `slice-13.json`
- Each chunk agent reads its dedicated slice file
- Per-chunk output unchanged
- **Pro:** one parse pass; chunk agents see only their own data
- **Con:** extra ~14 files on disk

**Recommendation: Option A.** Smaller blast radius. The redundant parses are seconds of overhead, not a real cost.

Either option also fixes a latent risk in `aggregate:final` where the agent reads the merged chunks from disk (currently fine because it never tries to return them through structured output — it writes directly to disk and only returns counts).
